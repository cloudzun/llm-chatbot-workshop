## 一、课程概览

### 1.1 教学目标

完成本工作坊后，学员应当能够：

1. **理解 LLM 应用的核心技术栈**：API 调用、流式输出、RAG、Function Calling
2. **独立完成一个可运行的 AI chatbot**，从零到具备天气/新闻/搜索能力
3. **掌握 Streamlit 的执行模型**，特别是 `session_state` 的必要性
4. **用 AI 编程助手（OpenCode）完成功能开发**，而不是手写所有代码
5. **具备基本的问题排查能力**，能读懂报错并知道如何处置

### 1.2 适合学员群体

| 群体 | 背景 | 预期完成情况 |
|------|------|------------|
| 零代码背景 | 会用电脑，没写过代码 | 完成 Phase 1~3，有 OpenCode 辅助下 Phase 4 |
| 初级开发者 | Python 入门水平 | 完成全部 6 个 Phase |
| 有经验开发者 | Python 熟练，想了解 LLM 应用 | 快速过理论，重点在 Phase 3~6 设计思路 |

### 1.3 课程结构选项

根据时间长度，推荐以下安排：

| 形式 | 时长 | 覆盖 Phase |
|------|------|----------|
| 半日工作坊（精华版） | 3~4 小时 | Phase 0~3 |
| 全日工作坊（标准版） | 6~8 小时 | Phase 0~6 |
| 两日密集营 | 12~16 小时 | Phase 0~6 + 定制项目 |

## 三、各阶段教学详解

### Phase 0：环境准备（15~30 分钟）

**教学目标**：让所有学员的环境准备好，一起进入下一阶段。

**讲师脚本**：

1. 展示安装完成的效果（`python --version` 输出、`streamlit hello` 界面）
2. 演示注册硅基流动账号的操作步骤（同步操作，学员跟做）
3. 演示创建 `.streamlit/secrets.toml`，填入 API Key
4. 带领所有人确认环境没问题

**常见阻碍与应对**：

| 问题 | 应对 |
|------|------|
| Windows 没有将 Python 加入 PATH | 演示如何重新安装勾选 "Add to PATH"，或手动添加 |
| 公司机器无法安装软件 | 使用事先准备好的便携版或虚拟环境 |
| API Key 注册收不到验证码 | 备用：讲师提供专门用于教学的临时 Key（有 Rate Limit，仅供课堂） |

**推进信号**：所有学员的 `streamlit hello` 能打开浏览器，且有 API Key 可用。

---

### Phase 1：基础聊天（45~60 分钟）

**教学目标**：学员第一次让 AI 通过自己写的程序"说话"。

**重点讲解（15 分钟）**：

> ⚠️ **本阶段最重要的知识点：Streamlit 执行模型**
>
> 建议用以下方式让学员直观感受到这个特性：
>
> ```python
> # 现场演示：写这段代码让学员观察
> import streamlit as st
> import random
>
> n = random.randint(1, 100)    # 每次交互会重新生成
> st.write(f"本次执行随机数: {n}")
>
> if st.button("点我"):
>     st.write("你点了按钮！")   # 点击后脚本重新执行，n 会变
> ```
>
> 然后说明：这就是为什么要用 `session_state` 保存状态。

**动手实操（30 分钟）**：

1. 带学员进入 `phase-1/starter/` 目录
2. 演示如何把 OpenCode 的指令（第一步）复制粘贴给 OpenCode
3. 学员跟做，助教巡场协助
4. 演示运行效果（`streamlit run app.py`）

**验收（5~10 分钟）**：

- 挑 2~3 名学员演示自己的界面
- 带全体学员做验收清单（学生教材 1.4 节）

**课堂互动建议**：

- 让学员问 AI 一个他们真正好奇的问题，分享有趣的回答
- 提问：「为什么清空对话后，之前的历史消失了？session_state 是跨浏览器会话持久的吗？」

---

### Phase 2：参数调优（30~45 分钟）

**教学目标**：学员能直观理解 temperature 和 system prompt 对 AI 行为的影响。

**重点讲解（10 分钟）**：

- temperature 可以用"创意旋钮"类比
- 演示极端值：`temperature=0` 问同一问题三次（结果相同），`temperature=1.5` 问三次（不同）
- System Prompt 演示：把 prompt 改成「只能用 emoji 回答」，效果直观有趣

**动手实操（20 分钟）**：

1. 进入 `phase-2/starter/`，或在 Phase 1 的基础上修改
2. OpenCode 指令（教材 2.2 节）
3. 带学员做「实验 2：System Prompt 的魔力」——把各自聊天机器人改成不同角色

**课堂互动建议**：

- 挑战：「用一段 System Prompt，让 AI 变成只研究易速鲜花产品的花艺专家，不回答其他问题」
- 提问：「如果 System Prompt 告诉 AI '你是人类，不是 AI'，AI 会听吗？」（答案：会在一定程度上遵从，但模型可能仍会在某些情形下"打破"）

---

### Phase 3：RAG 知识库（60~90 分钟）

**教学目标**：学员理解 RAG 的价值，并能把自己的文档接入 AI。

**重点讲解（20~25 分钟）**：

1. **RAG vs 纯模型**：演示同一个关于文档内容的问题：
   - 不开 RAG：AI 胡说或说不知道
   - 开 RAG（构建了索引）：AI 准确引用文档内容

2. **嵌入向量的直觉**：
   - 用「意思相近的词向量距离近」这个直觉，不需要讲公式
   - 可以用 t-SNE 可视化图（若有）直观展示

3. **分块的必要性**：展示如果整篇文档不分块，单次请求 Token 会超出限制

**动手实操（40~60 分钟）**：

这是本课程最复杂的阶段，分四步完成。

> ⚠️ **讲师注意事项**：
>
> - 构建索引需要调用嵌入 API（消耗额度，约 100~500 个 Token）
> - 首次构建 `oneflower/` 索引通常需要 30~60 秒
> - 学员容易在软链接/复制 oneflower 目录这步遇到权限问题（尤其 Windows）

**关于 oneflower/ 目录**：

演示如何从 `python/oneflower/` 创建软链接到各 phase 的 completed 目录：

```cmd
# Windows（管理员终端）
cd python\phases\phase-3\completed
mklink /D oneflower ..\..\..\oneflower
```

如果学员的电脑没有管理员权限创建符号链接，直接复制也可以：

```cmd
xcopy /E /I ..\..\..\oneflower oneflower
```

**验收要点**：

- 能看到「已索引 XX 个片段」的成功提示
- 关于「花语」的问题，RAG 模式下能引用文档内容并显示引用来源

**课堂互动建议**：

- 让学员添加自己带来的一份文档（如工作手册、产品文档），构建自己的专属知识库
- 测试有趣的边界：「如果文档里没有这个信息，AI 会怎么回答？」

---

### Phase 4：Function Calling（45~60 分钟）

**教学目标**：学员理解 Function Calling 的调用机制，能让 AI 使用外部工具获取实时信息。

**重点讲解（15 分钟）**：

用"两次调用"时序图讲解：

```
用户: "北京天气?"
     ↓
第 1 次调用 (非流式) ──→ AI 返回: "我要调用 get_weather(city='Beijing')"
     ↓
Python 代码: 真正发 HTTP 请求到 wttr.in
     ↓
第 2 次调用 (流式) ──→ AI 返回: "北京今天晴，28°C，带把伞..."
```

强调：**AI 自己不发网络请求**，它只是"提议"调用函数，Python 代码才是真正执行的一方。

**动手实操（25~30 分钟）**：

1. `tools/weather.py` 的实现相对简单，可以让学员先自己尝试
2. `app.py` 的 Function Calling 逻辑是重点，用 OpenCode 指令（教材 4.2 节）完成
3. 带学员看 completed 版的代码结构，理解两次调用的 messages 拼接方式

**常见问题**：

- 网络访问 `wttr.in` 超时：确认网络通畅，或演示时先用 VPN
- AI 不触发工具：检查 TOOLS 中的 description 是否足够明确

---

### Phase 5：多工具协作（30~45 分钟）

**教学目标**：学员掌握多工具场景下的 description 设计原则。

**重点讲解（10 分钟）**：

- 展示两个 description 冲突时 AI 选错工具的例子
- 总结：description 要写"正面约束"（适合什么场景）和"负面约束"（不适合什么场景）

**动手实操（20 分钟）**：

新增 `tools/news.py` 和在 app.py 中注册新工具。相对简单，主要是把 Phase 4 的模式重复一次。

**课堂讨论**：

「如果工具的 description 写得很差，会发生什么？」——让学员故意写一个模糊的描述，观察 AI 的工具选择出错。

---

### Phase 6：联网搜索（30~45 分钟）

**教学目标**：学员完成完整的四工具 chatbot，掌握优雅降级的设计意识。

**重点讲解（10 分钟）**：

- 展示"多工具决策树"示意图（教材 6.1 节）
- 讲解"优雅降级"：DuckDuckGo 无结果时返回百度链接，而不是崩溃或返回空

**动手实操（20 分钟）**：新增 `tools/search.py` 并注册。

**课程总结（10 分钟）**：

带学员回顾六个 Phase 叠加的能力，对照「各阶段能力矩阵」（教材附录 A）。

**最终 Demo**：

每位学员演示自己完成的最终版本，展示至少三个工具各被触发一次的场景。

## 四、常见学员困难及应对

### 4.1 Streamlit 执行模型不理解

**症状**：学员写了普通变量存对话历史，发现每次消息后历史丢失，困惑。

**教学干预**：
1. 打开已出问题的代码，**故意不修复**，让学员观察每次交互后变量的值
2. 在代码里加 `st.write(len(messages))` 显示列表长度，演示它每次回到 0
3. 引入 `session_state`，再观察长度变化
4. 总结：「在 Streamlit 里，想要东西'活过'一次执行，就要放进 session_state」

### 4.2 OpenCode 生成的代码和参考代码不同

**症状**：学员的代码和 `completed/` 目录里的不一样，不确定对不对。

**教学干预**：
- 引导学员聚焦**功能**而不是**代码形式**：「它能运行吗？功能对吗？」
- 如果功能正确，代码不同是正常的，OpenCode 每次生成会有差异
- 如果功能不对，把问题描述加错误信息发给 OpenCode

### 4.3 RAG 阶段卡在构建索引

**症状**：点「构建索引」后没有反应，或报错，或成功但检索为空。

**排查流程**：
1. 检查 `oneflower/` 软链接是否正确（`ls oneflower/` 或 `dir oneflower\` 能看到文件）
2. 检查 secrets.toml 中 API Key 是否有效
3. 查看 Streamlit 终端输出的 Traceback（学员有时忽视终端）
4. 尝试只加载一个 TXT 文件测试

### 4.4 工具调用不触发

**症状**：问「北京天气」，AI 直接用知识回答，没有触发 `get_weather`。

**排查**：
1. 检查 TOOLS 列表是否正确传入 `client.chat.completions.create`（`tools=TOOLS`）
2. 检查工具的 `description` 是否清晰
3. 确认模型支持 Function Calling（DeepSeek-V3 支持，某些 Qwen 小模型可能不稳定）
4. 换成 `deepseek-ai/DeepSeek-V3` 重试

### 4.5 第二次流式调用不显示（或没有文字）

**症状**：工具被触发、天气卡片出现，但后面没有文字总结。

**排查**：
1. 检查第二次调用的 `messages_for_second` 是否包含了 tool 角色的消息
2. 确认 `response: str = st.write_stream(stream)` 在 `with st.chat_message("assistant"):` 块内
3. 打印 `messages_for_second` 看结构是否正确

## 六、评估与反馈

### 6.1 即时评估方式

每个 Phase 结束后快速举手调查（或用 Slido/Mentimeter）：

- 「完全跑通了」✋
- 「跑通了但不太理解原理」✋

对「还没跑通」的学员：助教立即上前辅助，讲师可暂停进度或让跑通的学员互助。

### 6.2 知识验收（每章末尾）

教材中每章都有「知识自测」题目，可用于：

- **口头问答**：随机抽查 1~2 道，不点名回答
- **小组讨论**：两人一组互相讲解，讲清楚再继续
- **不用于正式考试**（工作坊性质不需要打分）

### 6.3 反馈收集

课程结束后发送问卷（建议用腾讯问卷或金数据）：

```
1. 你在哪个阶段收获最大？（单选）

4. 课程内容难度是否合适？（偏难 / 刚好 / 偏易）
5. 你会把这个项目继续发展下去吗？（是/否/不确定）
6. 最希望改进的一点是？（开放填写）
```

## 八、讲师 FAQ

**Q: 学员的 API 额度不够用了怎么办？**

A: 硅基流动的免费额度通常够一天的课程使用。如果不够，可以让学员之间临时共用一个 Key（注意 Rate Limit，不能超过并发限制）。更好的方案是提前为每位学员申请试用额度，或用教学用共享账户。

**Q: 某个 RSS 新闻链接被防火墙阻断怎么办？**

A: `news.py` 已内置 fallback 机制，RSS 失败时返回静态示例数据，课堂演示不受影响。可以提前测试哪些 RSS URL 在教学环境可访问，并在课前更新为可用的 URL。

**Q: DuckDuckGo API 返回总是空？**

A: DDG Instant Answer 对部分查询（尤其中文、热点事件）返回为空是正常的。fallback 链接（百度/必应）会自动出现。这本身也是教学点：真实项目需要准备降级方案。

**Q: 要不要讲 MCP（Model Context Protocol）？**

A: 本课程的"MCP"章节（原标题 Phase 5）实际上是通过 RSS + Function Calling 实现的 news 工具，已简化掉 Python MCP SDK 的 asyncio 复杂性。如果有学员问到正式的 MCP 协议，可以说「官方 MCP 需要处理 asyncio 事件循环兼容问题，这是进阶内容，本课程用更简单的实现达到了同样效果」。

**Q: 有学员想在自己的机器上部署，可以吗？**

A: 可以，直接复制代码，修改 `.streamlit/secrets.toml`，运行 `streamlit run app.py` 即可。Streamlit Cloud 免费托管也是一个选择。注意 `oneflower/` 知识库文件需要一起带走。

**Q: 能把 `BAAI/bge-m3` 换成其他嵌入模型吗？**

A: 可以。硅基流动支持其他嵌入模型，只需修改 `embeddings.py` 中的 `model` 参数。不同模型的向量维度可能不同，需要确认 vectorstore.json 中的向量维度一致。
