## 总体节奏

每个阶段约 **45 分钟**，结构固定：

| 环节 | 时长 | 讲师动作 |
|------|------|---------|
| 开场演示 | 2 分钟 | 展示本阶段完成后的效果（运行参考代码） |
| 概念讲解 | 10 分钟 | 结合教科书讲解核心概念 |
| 带练示范 | 5 分钟 | 在 OpenCode 中现场输入提示词，带学员看代码生成过程 |
| 学员实操 | 20 分钟 | 学员按教科书指引，用 OpenCode 自己操作 |
| 验证答疑 | 8 分钟 | 学员展示结果，集中解答 |

## 通用应急处理手册

在任何阶段，遇到以下问题时：

| 问题 | 标准应对 |
|------|---------|
| AI 生成的代码报错 | 「把报错信息粘贴给 OpenCode：*帮我修这个错误：[错误信息]*」 |
| AI 生成的代码跟参考不一样 | 正常！只要运行成功、功能正确就对了。强调：没有唯一正确答案 |
| API Key 失效 | 登录 siliconflow.cn → 账户设置 → 创建新密钥 → 更新 `.env` |
| 端口被占用 | 在 `.env` 中将 `PORT=3000` 改为 `PORT=3001`，重启 |
| npm 包安装失败 | 切换镜像：`npm config set registry https://registry.npmmirror.com`，重新 install |
| OpenCode 长时间不返回 | 重新输入，提示词加上「请只输出代码，不要解释」 |

## Phase 1：基础聊天

### 学习目标
学员能说出：「LLM API 是一个接受问题返回答案的服务，后端充当代理把前端请求转发过去，用 SSE 实现逐字显示效果。」

### 开场演示（2 分钟）
运行 `node phases/phase-1/completed/server.js`，打开浏览器，输入「你好，介绍一下你自己」，让学员看到文字逐字出现、底部 Token 计数更新。

### 概念讲解提纲（10 分钟）

1. **LLM API 是什么**（3 分钟）
   - 类比：像一个智慧的「问答机」，输入消息列表，输出回答
   - 关键：`messages` 数组里的角色：`system`（人设）、`user`（问题）、`assistant`（回答）
   - 画图：[前端] → POST → [后端] → API Call → [硅基流动] → response → [后端] → SSE → [前端]

2. **为什么要「代理」**（2 分钟）
   - API Key 不能暴露在前端（会被扒取）
   - 后端是「安全门」

3. **SSE 是什么**（3 分钟）
   - 类比：打电话时对方边想边说，不是想完了再说
   - 技术上：HTTP 长连接，服务器持续推送 `data: {...}\n\n`

4. **Token 是什么**（2 分钟）
   - 类比：中文约 1.5 字/token，英文约 0.75 词/token
   - 费用的计量单位

### 带练示范（5 分钟）
打开 OpenCode，现场输入 `docs/engineering/construction-manual.md` 里 Phase 1 的第二步提示词（后端部分），让学员看到 OpenCode 生成 `server.js` 的过程。**不要运行，只是展示流程。**

### 学员实操（20 分钟）
学员按教科书 Phase 1 章节，自己在 OpenCode 中完成三步：初始化项目 → 后端 → 前端。

**讲师巡场重点**：
- 确认学员的 `.env` 中 `LLM_API_KEY` 已填入真实 Key
- 确认 `package.json` 有 `"type": "module"`（否则 `import` 会报错）
- 确认执行了 `npm install`

### 验证答疑（8 分钟）
验收标准（让 2-3 名学员展示）：
- [ ] `npm start` 输出「服务已启动」
- [ ] 浏览器能看到聊天界面
- [ ] 输入消息后 AI 有流式回复
- [ ] 底部 Token 数字有变化

**高频问题**：
- *「AI 只生成了一个文件，其他文件在哪里？」* → 分步骤，分开提示
- *「前端界面空白」* → 检查 `public/` 路径是否正确，`express.static` 是否指向正确目录

## Phase 2：参数调优面板

### 学习目标
学员能亲身体验：同一问题在 temperature=0 和 temperature=1.5 的不同输出，并能解释「为什么调参数有效」。

### 开场演示（2 分钟）
运行 Phase 2 参考代码，展示：展开参数面板 → 调 temperature=0 → 问「写首诗」→ 再调 temperature=1.5 → 同样的问题，对比输出差异。

### 概念讲解提纲（10 分钟）

1. **Temperature 是什么**（3 分钟）
   - 类比：温度控制「随机性」，像骰子的偏向性
   - 0 = 永远选最高概率词（确定性最强）
   - 2 = 各词被选到的概率趋于均等（最随机）
   - 实用：知识问答用 0.2-0.4，创作用 0.7-1.0

2. **Top-P 是什么**（2 分钟）
   - 核采样：只从累计概率达到 P 的候选词中选择
   - 与 temperature 通常只调一个，不同时调两个

3. **System Prompt 是什么**（3 分钟）
   - 「潜台词」，对话全程生效，用户看不到
   - 可以设定语气、角色、约束
   - 演示：system prompt 设为「你是一位严厉的老师，回答必须简洁，不超过30字」

4. **Max Tokens 的意义**（2 分钟）
   - 控制单次回复最大长度，节约费用

### 带练示范（5 分钟）
现场用 OpenCode 添加参数面板的 HTML 部分（`docs/engineering/construction-manual.md` Phase 2 提示词），展示 OpenCode 如何修改现有代码。

### 学员实操（20 分钟）
基于 Phase 1 已完成的代码继续修改，添加参数面板。

**讲师巡场重点**：
- 确认前端参数是通过 POST body 中的 `params` 对象传递
- 确认后端有 `parseFloat(temperature)` 类型转换（容易漏）

### 对比实验（嵌入实操环节）
让学员做记录表：

| 问题 | Temperature | 回答（关键词） |
|------|------------|--------------|
| 讲个故事 | 0 | |
| 讲个故事 | 0.7 | |
| 讲个故事 | 1.5 | |

### 验证答疑（8 分钟）
- [ ] 参数面板能展开/折叠
- [ ] temperature=0 时多次问同一问题，回答几乎一致
- [ ] 修改 system prompt 后 AI 风格变化

## Phase 3：RAG 知识问答

### 学习目标
学员能用自己的话解释：「RAG 就是先搜索，再把搜索结果塞进提示词，让 AI 基于这些内容回答。」

### 开场演示（2 分钟）
运行 Phase 3 参考代码，先构建索引，然后：
- 关闭知识库模式，问「易速鲜花的退换货政策是什么」→ AI 说不知道
- 开启知识库模式，同样问题 → AI 准确引用文档回答，底部显示来源

### 概念讲解提纲（10 分钟）

1. **RAG 解决什么问题**（2 分钟）
   - LLM 的知识有截止日期，也没有你的私有数据
   - RAG = Retrieval Augmented Generation，检索增强生成

2. **RAG 的工作流程**（4 分钟）
   - 画图：
     ```
     提问 → [嵌入] → 向量 → 余弦相似度 → 最相关片段
                                              ↓
                                     拼进 System Prompt
                                              ↓
                                       LLM 回答
     ```
   - 嵌入：把文字变成数字向量，语义相似的词/句向量距离近
   - 余弦相似度：衡量两个向量方向有多相似（-1 到 1）

3. **为什么不直接把全文档给 LLM**（2 分钟）
   - Context 长度限制（几千到几十万 token）
   - 成本：token 越多越贵
   - 效果：太长的 context LLM 容易忽略中间内容

4. **[进阶] 重排序是什么**（2 分钟）
   - 嵌入检索是粗排（速度快），重排序是精排（精度高）
   - 两阶段：先取 Top-10，再重排取 Top-3

### 带练示范（5 分钟）
现场演示：用 OpenCode 创建 `rag/loader.js`，展示文档切分逻辑。

### 学员实操（20 分钟）
分 5 个子步骤（可以让学员分组，每组专注一个模块）。

**讲师巡场重点**：
- 确认 `.env` 中 `KNOWLEDGE_DIR=./OneFlower/OneFlower`（路径正确）
- 嵌入 API 调用要注意 `input` 字段（不是 `inputs`）
- 构建索引后 `rag/vectorstore.json` 文件应该出现

### 验证答疑（8 分钟）
- [ ] 点「构建索引」，控制台显示片段数（>0）
- [ ] 开启知识库模式，问易速鲜花相关问题有准确回答
- [ ] 关闭后同样问题 AI 说不确定

## Phase 4：Function Calling — 天气查询

### 学习目标
学员能描述「两次调用」模式：「第一次问 LLM 该不该调用工具，如果要调用，执行完工具再第二次调用 LLM 生成回答。」

### 开场演示（2 分钟）
运行 Phase 4 参考代码，问「北京今天天气怎么样」，展示天气卡片出现；再问「写首关于秋天的诗」，展示正常聊天（不触发工具）。

### 概念讲解提纲（10 分钟）

1. **Function Calling 解决什么问题**（2 分钟）
   - LLM 只能处理文本，不能直接访问外部系统
   - Function Calling 让 LLM「委托」外部代码完成任务

2. **Function Calling 的工作流程**（5 分钟）
   - 画图：
     ```
     用户: 北京天气? 
       ↓
     [后端] POST /chat/completions + tools 定义
       ↓
     LLM: 我要调用 get_weather(city="Beijing")
       ↓
     [后端] 执行天气 API → 得到数据
       ↓
     [后端] POST /chat/completions + 工具结果
       ↓
     LLM: 北京今天晴天，23℃...
     ```
   - 关键：LLM 不「调用」工具，只是「声明」要调用哪个工具、传什么参数

3. **工具定义中 description 的重要性**（3 分钟）
   - LLM 通过 description 决定调用哪个工具
   - 越精确越好，要写清楚「什么时候调」

### 带练示范（5 分钟）
现场演示：OpenCode 创建 `tools/weather.js`，然后修改 `server.js` 添加工具定义。

### 学员实操（20 分钟）
**讲师巡场重点**：
- 两次调用逻辑是难点，如果 OpenCode 生成的逻辑有误，让学员粘贴报错信息给 OpenCode 修
- 天气 API (`wttr.in`) 需要网络访问，若有防火墙问题，可临时 mock 一个返回值

### 验证答疑（8 分钟）
- [ ] 问天气，看到真实数据（非 mock）
- [ ] 问非天气问题，正常聊天
- [ ] 天气卡片样式正常显示

## Phase 5：MCP 对接 — 新闻阅读

### 学习目标
学员能解释 MCP 协议的核心价值：「一套标准化的方式让 AI 应用接入外部工具，像 USB 接口一样，开发一次到处用。」

### 开场演示（2 分钟）
运行 Phase 5 参考代码，问「今天科技类新闻有哪些」，展示新闻列表出现；再验证天气功能仍然正常。

### 概念讲解提纲（10 分钟）

1. **MCP 是什么**（3 分钟）
   - Model Context Protocol：Anthropic 提出的开放协议
   - 类比：AI 的「USB 标准」，任何工具都能通过 MCP 接入任何 AI 应用
   - 之前每个 AI 应用需要为每个工具写定制代码，MCP 统一了接口

2. **MCP 架构**（4 分钟）
   - 画图：
     ```
     [AI 应用/Host] ←→ [MCP Client] ←stdio→ [MCP Server] → [外部服务]
     ```
   - MCP Server：封装外部能力，暴露工具列表
   - MCP Client：调用 Server 工具
   - stdio：进程间通信，通过标准输入输出传递消息

3. **feed-mcp 是什么**（3 分钟）
   - 一个现成的 MCP Server，封装了 RSS 订阅能力
   - 通过 `npx` 按需启动，不需要提前安装
   - RSS：网站发布更新的标准格式（新闻、博客都用）

### 带练示范（5 分钟）
现场演示：安装 MCP SDK，创建 `mcp/mcp-config.json`，让学员看 MCP Client 初始化代码。

**⚠️ 注意**：feed-mcp 版本更新快，若出现兼容性问题，提前准备备用方案（见通用应急处理手册）。

### 学员实操（20 分钟）
**讲师巡场重点**：
- `@modelcontextprotocol/sdk` 安装有时较慢，提前预装
- MCP Server 启动需要 npx，检查学员机器是否有网络权限
- 注意 feed-mcp 实际工具名称（用 `tools/list` 验证）

### 验证答疑（8 分钟）
- [ ] 问新闻，返回条目（不是空列表）
- [ ] 天气查询仍然正常
- [ ] 新闻卡片样式正常

## Phase 6：DuckDuckGo 搜索 + 多工具编排

### 学习目标
学员能完成「多工具自动分派」测试，理解工具 `description` 的区分度决定 AI 的工具选择准确性。

### 开场演示（2 分钟）
运行 `npm run solution`（Phase 6 完整版），展示：
- 「北京天气」→ 天气卡片
- 「今日新闻」→ 新闻列表
- 「什么是量子计算」→ DuckDuckGo 摘要
- 「写首诗」→ 直接回答（不调用任何工具）

### 概念讲解提纲（10 分钟）

1. **多工具编排的挑战**（3 分钟）
   - 工具足够多时，LLM 靠 `description` 判断用哪个
   - 描述模糊 → 误触发；描述精确 → 正确分派

2. **DuckDuckGo Instant Answer API**（3 分钟）
   - 完全免费、无需 Key
   - 返回摘要/定义，非完整搜索结果列表
   - 适合「什么是 X」类查询

3. **RAG + 工具如何协同**（4 分钟）
   - RAG 解决私有知识问题（知道自家知识库）
   - 工具解决实时信息问题（天气/新闻/新知识）
   - 完整优先级：RAG 开关 → tool_calls 触发 → 直接回答

### 带练示范（5 分钟）
现场用 OpenCode 添加 `search_web` 工具定义和 handler。

### 多工具测试矩阵（嵌入实操环节）

让学员测试并填写下表：

| 问题 | 预期工具 | 实际行为 | 是否符合 |
|------|---------|---------|---------|
| 北京今天天气 | get_weather | | |
| 今日科技新闻 | get_news | | |
| 什么是区块链 | search_web | | |
| 易速鲜花的退换货政策 | RAG | | |
| 给我写首诗 | 无（直接回复） | | |

### 验证答疑（8 分钟）
- [ ] 上表全部填写完成
- [ ] 至少 4/5 行「是否符合」为✅
- [ ] 四种工具均成功调用过一次
